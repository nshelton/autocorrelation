# v3.1 — ParamStore + tweakpane Design

**Date:** 2026-04-28
**Status:** Design approved, awaiting plan

## Goal

Make the analysis parameters that are currently hard-coded constants user-adjustable at runtime via a tweakpane-backed UI, with values persisted to `localStorage` across sessions. Ship the infrastructure — `ParamStore`, schema-driven UI, worklet reconfiguration channel — alongside exposing the five existing constants. New DSP features (autogain, additional smoothing modes, etc.) are explicitly deferred to a later sub-version so v3.1 stays focused on the framework.

## Architecture

A single `ParamStore` on the main thread is the source of truth. Each parameter is registered with a schema (key, label, default, kind, range/options, reconfig flag). On registration, the store loads the value from `localStorage` or falls back to the default. Writes go through `set()`, which validates against the schema, persists, and notifies subscribers.

A `ParamPanel` wraps a `tweakpane` `Pane`, building widgets from the registered schemas. Tweakpane `change` events call `ParamStore.set()`; store subscriptions update tweakpane bindings in the reverse direction so `reset()` and other programmatic changes refresh the UI.

A `WorkletBridge` subscribes to the store and translates parameter changes into messages for the audio worklet. Reconfig parameters (window size, RMS history length) cause the worklet to drop and recreate its `Dsp` instance and post a `"configured"` message back with the new buffer sizes; the App responds by disposing the old `LineRenderer` instances and constructing new ones at the new sizes. Hot parameters (hop size, smoothing α, dB floor) flow as `param` messages and are applied via setters on the `Dsp` instance or instance variables on the worklet.

## Tech Stack

- **`tweakpane`** (~40 KB gzipped) — added as a runtime `dependency` in `package.json`. Provides the panel UI (sliders, dropdowns, folders, reset button).
- **TypeScript / Vite** — no other new dependencies.
- **Rust DSP** — no new dependencies. Two existing constants (`SMOOTHING_ALPHA`, `DB_FLOOR`) become struct fields with setters; `RMS_HISTORY_LEN` becomes a constructor argument; `RMS_ACF_LEN` becomes a derived value (`rms_history_len / 2`).
- **Vitest + happy-dom** — already configured; happy-dom supplies `localStorage` for `ParamStore` tests.

## Components

### `ParamStore`

```ts
export type ParamValue = number;

export type ParamSchema = {
  key: string;
  label: string;
  default: ParamValue;
  reconfig: boolean;
} & (
  | { kind: "discrete"; options: ParamValue[] }
  | { kind: "continuous"; min: number; max: number; step: number }
);

export class ParamStore {
  private values = new Map<string, ParamValue>();
  private schemas = new Map<string, ParamSchema>();
  private subscribers = new Set<(key: string, value: ParamValue) => void>();
  private storageKey = "autocorrelation.params.v1";

  register(schema: ParamSchema): void;
  get(key: string): ParamValue;
  set(key: string, value: ParamValue): void;
  subscribe(fn: (key: string, value: ParamValue) => void): () => void;
  reset(): void;
  getAll(): Record<string, ParamValue>;
  schemasInOrder(): ParamSchema[];
}
```

- **`register`** — installs the schema, then either loads the value from `localStorage` (under `storageKey`) or falls back to `schema.default`. The first registration also reads the entire localStorage entry into a memo so subsequent registrations are O(1).
- **`get`** — throws if the key isn't registered; this is a programmer error.
- **`set`** — validates: discrete values must be in `options`; continuous values must be in `[min, max]`. On success, writes the new value, persists the entire `values` map to localStorage as JSON, and notifies subscribers. On validation failure, no-op (this is also a programmer error; tweakpane enforces ranges in the UI).
- **`subscribe`** — returns an unsubscribe function. Subscribers receive `(key, newValue)` synchronously inside `set()`.
- **`reset`** — wipes `localStorage[storageKey]`, repopulates `values` from each schema's `default`, fires a subscriber notification for each changed key.
- **Storage key** carries a `v1` suffix so future schema-incompatible changes can bump to `v2` without orphaning state.

### `ParamPanel`

```ts
import { Pane } from "tweakpane";

export class ParamPanel {
  private pane: Pane;

  constructor(store: ParamStore, container?: HTMLElement) {
    this.pane = new Pane({ title: "Analysis", container });
    const folder = this.pane.addFolder({ title: "DSP" });

    const bindings: Record<string, ParamValue> = {};
    for (const schema of store.schemasInOrder()) {
      bindings[schema.key] = store.get(schema.key);
      const widget = schema.kind === "discrete"
        ? folder.addBinding(bindings, schema.key, {
            label: schema.label,
            options: Object.fromEntries(schema.options.map(v => [String(v), v])),
          })
        : folder.addBinding(bindings, schema.key, {
            label: schema.label,
            min: schema.min,
            max: schema.max,
            step: schema.step,
          });
      widget.on("change", (e: { value: ParamValue }) => store.set(schema.key, e.value));
    }

    store.subscribe((key, value) => {
      if (bindings[key] !== value) {
        bindings[key] = value;
        this.pane.refresh();
      }
    });

    this.pane.addButton({ title: "Reset to defaults" }).on("click", () => store.reset());
  }

  dispose() {
    this.pane.dispose();
  }
}
```

The panel anchors to its default location (top-right). The FPS overlay moves to top-left to free that space.

### `WorkletBridge`

```ts
export class WorkletBridge {
  constructor(private store: ParamStore, private port: MessagePort) {
    store.subscribe((key) => this.handleChange(key));
  }

  bootstrap(): void {
    this.port.postMessage({
      type: "configure",
      windowSize: this.store.get("dsp.windowSize"),
      rmsHistoryLen: this.store.get("dsp.rmsHistoryLen"),
    });
    for (const k of ["hopSize", "smoothingAlpha", "dbFloor"] as const) {
      let value = this.store.get(`dsp.${k}`);
      if (k === "hopSize") {
        value = Math.min(value, this.store.get("dsp.windowSize"));
      }
      this.port.postMessage({ type: "param", key: k, value });
    }
  }

  private handleChange(key: string): void {
    if (key === "dsp.windowSize" || key === "dsp.rmsHistoryLen") {
      this.port.postMessage({
        type: "configure",
        windowSize: this.store.get("dsp.windowSize"),
        rmsHistoryLen: this.store.get("dsp.rmsHistoryLen"),
      });
    } else if (key.startsWith("dsp.")) {
      const hotKey = key.slice("dsp.".length);
      let value = this.store.get(key);
      if (hotKey === "hopSize") {
        value = Math.min(value, this.store.get("dsp.windowSize"));
      }
      this.port.postMessage({ type: "param", key: hotKey, value });
    }
  }
}
```

`bootstrap()` is called once after the worklet is constructed and before the `LineRenderer`s are built, so the worklet has a known good configuration before audio starts flowing. The hop-size clamp lives in the bridge (not the UI or store) so the UI can present "all four hop options always" without the user needing to reason about constraint ordering.

### Worklet (`src/audio/dsp-worklet.ts`) — message protocol

**Inbound** (main → worklet):
```ts
type WorkletInbound =
  | { type: "configure"; windowSize: number; rmsHistoryLen: number }
  | { type: "param"; key: "hopSize" | "smoothingAlpha" | "dbFloor"; value: number };
```

**Outbound** (worklet → main):
```ts
type WorkletOutbound =
  | { type: "features"; waveform: Float32Array; spectrum: Float32Array; rms: Float32Array;
      bufferAcf: Float32Array; rmsAcf: Float32Array }
  | { type: "configured"; waveformLen: number; spectrumLen: number;
      bufferAcfLen: number; rmsLen: number; rmsAcfLen: number };
```

`"features"` is unchanged from v3. `"configured"` is new and is sent immediately after a successful Dsp recreate.

**Configure handler:** drops the existing Dsp via `dsp.free()` (wasm-bindgen-generated finalizer), constructs a new `Dsp(windowSize, rmsHistoryLen)`, reallocates the sliding `window: Float32Array` to the new `windowSize`, resets `hopCounter = 0`, clamps the current `hopSize` to `≤ windowSize`. Then posts `"configured"` with the new buffer lengths so the App can rebuild renderers.

**Param handler:** hop size updates `this.hopSize` directly (clamped); smoothing α and dB floor go through new Rust setters (`set_smoothing_alpha`, `set_db_floor`).

### Rust DSP (`crates/dsp/src/lib.rs`)

**Signature change:** `Dsp::new(window_size: usize, rms_history_len: usize)`.

**Field changes:**
- `smoothing_alpha: f32` (was `const SMOOTHING_ALPHA = 0.2`)
- `db_floor: f32` (was `const DB_FLOOR = -100.0`)

**New methods:**
```rust
pub fn set_smoothing_alpha(&mut self, alpha: f32) {
    self.smoothing_alpha = alpha.clamp(0.0, 1.0);
}

pub fn set_db_floor(&mut self, floor: f32) {
    self.db_floor = floor.clamp(-200.0, 0.0);
}
```

**`process()` updates:** uses `self.smoothing_alpha` and `self.db_floor` in place of the removed consts. No other behavior changes.

**Removed:** `const SMOOTHING_ALPHA`, `const DB_FLOOR`, `const RMS_HISTORY_LEN`, `const RMS_ACF_LEN`. The first two become fields; the second two become a constructor parameter and a derived expression.

### App.ts wiring

The boot sequence in `App.start()`:

1. `const paramStore = new ParamStore();`
2. `for (const s of analysisSchemas) paramStore.register(s);`
3. `const panel = new ParamPanel(paramStore);` (mounts the UI)
4. Construct the audio source, the worklet node.
5. `const bridge = new WorkletBridge(paramStore, node.port);`
6. `bridge.bootstrap();` (sends initial configure + 3 param messages)
7. Wait for the worklet's first `"configured"` message, prime the store with the announced sizes, construct the five `LineRenderer`s, scene-add them.
8. Set `node.port.onmessage` to the steady-state handler, which dispatches on `msg.type` (`"features"` updates the store; `"configured"` triggers the dispose+rebuild path).
9. Start the render loop.

**`"configured"` handler:**
```ts
if (msg.type === "configured") {
  for (const line of [waveformLine, bufferAcfLine, spectrumLine, rmsLine, rmsAcfLine]) {
    line.dispose();
  }
  this.store.set("waveform", new Float32Array(msg.waveformLen));
  this.store.set("spectrum", new Float32Array(msg.spectrumLen));
  this.store.set("rms", new Float32Array(msg.rmsLen));
  this.store.set("bufferAcf", new Float32Array(msg.bufferAcfLen));
  this.store.set("rmsAcf", new Float32Array(msg.rmsAcfLen));
  // reconstruct LineRenderers (same layout/colors as v3) using the new lengths
  // ...
}
```

A short visible glitch (zero-line for one or two frames) on size change is acceptable.

### `LineRenderer.dispose()`

```ts
dispose(): void {
  this.geometry.dispose();
  this.material.dispose();
  this.object3d.parent?.remove(this.object3d);
}
```

This is the standard Three.js cleanup path. The dispose+rebuild approach intentionally avoids resizing `BufferAttribute`s in place, which was the source of the v1 WebGPU bug.

## Error Handling

- **Out-of-range `set()`:** silently no-ops with a `console.warn`. This is a programmer error since tweakpane enforces ranges in the UI.
- **Unknown key on `get()` / `set()`:** throws (programmer error).
- **Corrupt localStorage entry** (e.g., user-edited JSON, schema-incompatible old version): the JSON parse is wrapped in `try/catch`; on failure, treat the entry as missing and fall back to defaults. Do NOT delete the corrupt entry — leaving it in place lets the user recover or inspect; the next clean `set()` will overwrite it.
- **Worklet receives messages before WASM finishes loading:** the worklet's `boot()` is async (loads WASM + constructs `Dsp`), so a `configure` or `param` message can arrive before `this.dsp` exists. The worklet stores the most recent `configure` in `this.pendingConfigure` and the most recent value of each hot param in `this.pendingParams`. When `boot()` completes, it applies the pending configure (if any) by constructing the `Dsp` with those sizes — overriding any sizes the constructor used as defaults — then applies pending hot params via setters. If no `configure` arrived during boot, `boot()` constructs a default `Dsp(2048, 512)` so audio processing can begin even without a panel attached. Pending params are kept as "most recent" not as a queue: hot-param messages are idempotent and only the latest value matters.
- **Hop size > window size invariant:** clamped at the `WorkletBridge` send site. UI is permissive (lets you select any hop value).
- **`localStorage` unavailable** (Safari private browsing, file:// origin): wrap reads/writes in `try/catch`; the store works in-memory only. Values won't persist but the UI still functions.

### Manual acceptance checklist

- Open the app, click a start button, open the panel.
- Drag **smoothing α** to 0.95 — spectrum visibly smears slow. Drag to 0.05 — spectrum becomes jittery. Reload page — slider value persists, UI shows the persisted value.
- Click **window size = 512**. All five lines visibly change density (spectrum has 256 bins, buffer-ACF has 256 lags). FPS still >60.
- Click **window size = 4096**. Lines become denser. FPS still acceptable.
- Click **hop size = 256**. Spectrum updates faster; no errors in console.
- Click **dB floor = −60**. Spectrum dynamic range visibly tightens.
- Click **RMS history length = 256**. The RMS line shortens; RMS-ACF line shortens to 128 lags. No regression on spectrum or buffer-ACF.
- Click **Reset to defaults**. All sliders snap back, panel and worklet stay in sync, localStorage entry is empty (verify in devtools).
- Click each preset 1–6 — no regression on camera presets.
- FPS overlay is now top-left, panel is top-right; nothing overlaps.

## Out of Scope (Deferred)

- **New DSP features** (autogain / AGC, ACF smoothing, pre-emphasis, additional detrending modes, alternative ACF normalizations like NSDF) — listed as v3.2 candidates on the roadmap.
- **Shared message-shape types** between worklet and App for type-safe `postMessage` — pre-existing duplication that v3.1 extends; cleanup deferred.
- **Centralized constants module** for window/history/ACF sizes used by both worklet and App — same reasoning.
- **Decentralized "each module owns its params" architecture** — premature for one analysis module; revisit when a second module appears.
- **Visual / camera params** (line colors, baselines, heights) in the panel — out of scope for v3.1; the panel scope is "analysis" only.
